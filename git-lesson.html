Practical Git Lesson

Git Operations

Git records changes to a project as a sequence of named checkpoints. GitHub stores a remote copy of that history, but Git itself works locally on your computer.

The central idea: editing a file, staging a change, committing a snapshot, and pushing to GitHub are four separate operations.

Start Here

Turn an existing folder into a Git repository

Use Git Bash to begin tracking a folder that already contains your project files, then connect it to an empty GitHub repository when you are ready.

Before you begin

Create a new repository on GitHub without adding a README, license, or .gitignore. Keeping the GitHub repository empty prevents an unnecessary first-commit conflict.

1. Open the folder in Git Bash

In File Explorer, open the project folder, right-click an empty area, and choose Open Git Bash here. You can also navigate there manually:

cd "/c/path/to/your/project-folder"

2. Initialize the local repository

git init
git status

git init adds Git's hidden .git directory. Your existing project files remain where they are.

3. Create the first checkpoint

git add -A
git commit -m "Initial commit"

This stages the current folder contents and records the first local snapshot.

4. Connect the folder to GitHub

git branch -M main
git remote add origin https://github.com/USERNAME/REPOSITORY.git
git push -u origin main

Replace the example address with the URL of your empty GitHub repository. The -u option remembers the connection, so later you can normally use git push.

Existing Folder
git init
Initial Commit
Add Remote
Push to GitHub

Verify the connection

git remote -v
git status

The first command shows the GitHub address. The second confirms the current branch and whether the working folder is clean.

If Git asks for your identity

git config --global user.name "Your Name"
git config --global user.email "YOUR_GITHUB_EMAIL"

Run the commit command again after setting these values.

1. The Git model

Your project moves through a small set of states.

Working Folder
Staging Area
Local Commit
Push
GitHub

Working folder

The files you see in VS Code. Edits, new files, renames, and deletions begin here.

Staging area

A preparation zone where you choose which changes belong in the next snapshot.

Commit history

A chain of permanent local checkpoints with author, time, message, and unique ID.

Git and GitHub are not the same thing

Git manages project history on your computer. GitHub hosts a remote copy for backup, publishing, sharing, and synchronization.

2. Set your Git identity

Every commit records who created it.

Set your name

git config --global user.name "Kevin Linstrum"

This name is written into future commits on this computer.

Set your email

git config --global user.email "YOUR_GITHUB_EMAIL"

Using the email connected to GitHub helps associate commits with your account.

What --global means

The setting applies to all repositories on this computer.

Check the values

git config --global user.name
git config --global user.email

3. Inspect the repository

Run this whenever you are unsure what Git sees.

git status

git status

It shows the branch, new files, modified files, deleted files, staged changes, unstaged changes, and whether the local branch differs from GitHub.

4. Stage changes

Staging means selecting changes for the next snapshot.

Stage everything

git add -A

Stages new, modified, deleted, and renamed files.

Stage one file

git add filename.html

Use this when only one file belongs in the next commit.

Stage one folder

git add transistor-history/

Stages all changes inside that folder.

Why “add” includes deletions

git add really means “add this change to the next snapshot.” The change itself may be a deletion.

5. Create a commit

A commit records the staged changes as a named checkpoint.

Commit with a message

git commit -m "Remove unused source images"

The -m option supplies the message directly in the terminal.

A commit records

  • The staged changes
  • Your identity
  • The date and time
  • A unique ID
  • A link to the previous commit

A commit does not

  • Automatically update GitHub
  • Include unstaged changes
  • Require internet access

Good messages

  • Add transistor lesson 12
  • Fix lesson navigation links
  • Remove unused images

6. Push and pull

These commands synchronize your local repository with GitHub.

Push

git push

Sends local commits to GitHub.

Pull

git pull

Downloads and integrates newer commits from GitHub.

7. The normal working cycle

This is the practical VS Code sequence.

git pull
git status
git add -A
git status
git commit -m "Describe the work"
git push

Before editing

Pull so your local copy is current.

Before committing

Use status to verify exactly what is included.

After committing

Push so GitHub receives the checkpoint.

8. Separate unrelated changes

You do not have to commit everything at once.

Example

git add transistor-history/transistor-history-lesson12.html
git commit -m "Add transistor lesson 12"

git add images/
git commit -m "Reorganize image assets"

Why separate commits help

  • History is easier to understand.
  • Problems are easier to isolate.
  • One change can be reversed more safely.
  • Commit messages stay meaningful.

9. Basic recovery commands

Git is useful partly because mistakes can often be reversed.

Discard an unstaged edit

git restore filename.html

Restores the last committed version.

Unstage a file

git restore --staged filename.html

Keeps the edit but removes it from the next commit.

View recent commits

git log --oneline

Shows a compact commit history.

Safety rule

Before using commands containing reset, clean, or --force, verify exactly what they will do.

11. The Learning Library workflow

This is how Git fits your actual project.

1. Create

Edit lessons, images, and navigation.

2. Preview

Open the files locally and verify the result.

3. Commit

Package one coherent group of changes.

4. Push

Send the finished checkpoint to GitHub.

The deeper lesson

Git turns a folder of files into a documented sequence of decisions. The project contains not only its current state, but also a recoverable history of how it became what it is.
